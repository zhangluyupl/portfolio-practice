## 一、Context 的真实构成（每次对话烧多少 token）

Context = 所有发给模型的内容，分四层：

1. **System prompt**（OpenClaw 构造）
   - Tool list + descriptions
   - Skills list（name/description/location，非全文）
   - Workspace 路径、runtime 信息
   - 时间（UTC + 用户时区）
   - **Workspace bootstrap 文件**（AGENTS/SOUL/TOOLS/IDENTITY/USER/HEARTBEAT/BOOTSTRAP）

2. **Conversation history**（聊天记录）

3. **Tool calls + results**（工具执行输出）

4. **Attachments**（图片/音频/文件）

**关键细节：**
- Tool schemas（JSON）算入 context，即使你看不见
- 典型工具 schema：browser ~9.8K chars，exec ~6.2K chars
- 12 个 skills 的 skills list 约 2K chars overhead
- 每次 `/context detail` 可以看到各部分大小

**对我们有意义的数据：**
- Bootstrap 文件总上限：150K chars（我们 24K，安全）
- 单文件上限：20K chars（我们最大 8.6K 安全）
- Tool schemas 是隐藏的大户，不是 workspace 文件

## 二、Compaction 触发时会发生什么

**Compaction = 对话太长时，把历史压缩成摘要**

机制：
1. 触发时机：context 快满时自动，或 `/compact` 手动
2. 压缩时 OpenClaw **自动提醒 agent 存 memory**（防止上下文丢失）
3. 摘要写入 session transcript，原始历史保留在磁盘
4. 模型只看到摘要 + 近几轮完整消息

**对 agent 的实际影响：**
- Compaction 触发前，我应该主动把重要内容写 memory
- 压缩后早期对话的细节会丢失，只剩摘要
- 如果某个话题太早被压缩，可能之后无法引用细节

**防止丢失的方法：**
- 重要决策/洞见 → 立即写 MEMORY.md
- 不要依赖"对话里再翻出来"

## 三、Session Pruning vs Compaction

| | Pruning | Compaction |
|--|---------|------------|
| **做什么** | 裁剪旧的 tool results | 把历史对话总结成摘要 |
| **是否保存** | 否（仅在内存层面） | 是（写入 transcript） |
| **范围** | tool 输出结果 | 整个对话历史 |

两者互补——Pruning 保持 tool 输出精简，Compaction 处理对话历史积累。

## 四、Heartbeat 的完整机制

**默认配置：**
- 每 30 分钟一次（在主 session 运行）
- 每次都带完整对话历史（~100K tokens）
- Prompt：`Read HEARTBEAT.md if it exists... If nothing needs attention, reply HEARTBEAT_OK.`

**HEARTBEAT_OK 的边界：**
- 回复只有 HEARTBEAT_OK（前后均可）→ 不发消息，省 token
- 带了真实内容 → 正常发送
- 中间出现 HEARTBEAT_OK → 不触发特殊行为

**isolatedSession 模式：**
- 每次心跳开新 session，无历史包袱
- ~2-5K tokens/run vs 100K tokens（降幅巨大）
- delivery 路由仍用主 session 上下文

**lightContext 模式：**
- 只注入 HEARTBEAT.md，不注入其他 bootstrap 文件
- 适合"只做检查"的心跳任务

## 五、Cron 任务的三种执行模式

| 模式 | 走什么 | 上下文 |
|------|--------|--------|
| `main` | system event → 下一心跳 | 完整主 session 历史 |
| `isolated` | 独立 agent turn | 全新 session，无历史 |
| `current` | 绑定到创建时的 session | 维持上下文 |

**关键洞察：**
- 我们的 cron 任务全部用 `isolated`，每次都是全新 session
- 这意味着：凌晨·认知复盘 每次都不知道上次写了什么（除非自己读文件）
- 如果需要跨 run 维持上下文，应该用 `session:custom-id` 而非 `isolated`

**Cron 错误重试策略：**
- 瞬时错误（限流/网络/过载）→ 指数退避重试
- 永久错误（认证失败/配置错误）→ 直接禁用

## 六、Context Engine 机制

**Legacy engine（默认）：**
- Ingest：no-op
- Assemble：直接 pass-through
- Compact：调用内置 summarization

**Plugin engine 可以做什么：**
- 在 Assemble 阶段注入动态 system prompt addition
- 实现自定义压缩策略（DAG summaries、vector retrieval 等）
- 在 Ingest 阶段为每条消息建索引

**对我们直接的影响：**
- 如果安装 memory search 插件（embedding-based），assemble 时可以动态召回相关记忆
- 目前我们用的是 legacy engine，memory search 是 basic keyword 模式

## 七、System Prompt 的固定组成

每次运行，OpenClaw 构造的 system prompt 包含（按顺序）：

1. Tooling（工具列表 + 简短描述）
2. Safety（安全提醒）
3. Skills（可用 skills 的 name/description/location）
4. OpenClaw Self-Update（config.apply / update.run 说明）
5. Workspace（工作目录路径）
6. Documentation（本地 docs 路径 + 公共镜像）
7. **Project Context（workspace 文件注入）**
8. Sandbox（如启用）
9. Current Date & Time
10. Reply Tags
11. Heartbeats（心跳说明）
12. Runtime（host/OS/node/model/repo/thinking level）
13. Reasoning（推理可见性）

**Project Context 注入的文件（我们有的）：**
AGENTS.md / SOUL.md / TOOLS.md / IDENTITY.md / USER.md / HEARTBEAT.md / BOOTSTRAP.md / MEMORY.md

## 八、高价值行动项

1. **cron 任务之间需要上下文连续时** → 改用 `session:固定名称` 而非 `isolated`
   - 例：晨间·凌晨进化汇报 每次应该能读前一天写的 night-*.md

2. **`/context detail` 可以看到各部分 token 消耗** → 有需要时可以诊断

3. **Compaction 前 OpenClaw 会提醒 agent 存 memory** → 重要内容立即写文件，不要等

4. **Tool schemas 是隐藏的 context 大户** → 减少不必要的工具调用可以降低每次消耗
