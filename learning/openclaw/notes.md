## 核心模式一：任务系统思维（★★★★★）

**来源**：《研究完OpenClaw源码后，我的10条使用原则》

**核心观点**：OpenClaw 本质是任务调度系统，不是聊天助手。
给它模糊指令，它会在大空间里乱跑。

**操作方式**：每次会话 = 给它下一张任务单，定义清楚：
- 服务场景
- 可用工具
- 输入材料
- 期望输出
- 停止条件

**适用任务**：
- ✅ 有明确输入输出的知识工作（资料搜集、文档整理、方案初稿、代码排查）
- ✅ 需要工具参与的半自动执行（批量读文件、搜索项目文档）
- ❌ 模糊闲聊、模糊探索

**模式价值**：这与"致人而不致于人"高度契合——定义清晰的任务边界，就是掌握主动权的具体方式。

## 核心模式二：两段式调用（★★★★）

**来源**：《研究完OpenClaw源码后，我的10条使用原则》

**核心观点**：复杂任务不要让 Agent 同时做探索和总结，容易又长又散。

**操作方式**：
- **第一轮**：探索（搜集、扫描、初步归类、列出可能路径）
- **第二轮**：收口（只基于第一轮结果，输出判断、摘要、结论）

**适用场景**：
- 文章撰写：先调研再定稿
- 方案分析：先拆解再综合
- 项目研究：先扫描再判断

**模式价值**：这本身就是一个可迁移的认知工作流。写文章、做分析，都适用"先发散后收敛"。

## 核心模式三：最小必要上下文（★★★★）

**来源**：《研究完OpenClaw源码后，我的10条使用原则》+ Token 降本技巧

**核心观点**：上下文越长，token 消耗越高，模型注意力越被稀释。

**操作原则**：
- 长期规则文件只保留真正稳定的原则
- 一个文件只写一类事
- 背景材料尽量摘要化，不整篇原文硬塞
- 当前任务无关的资料，默认不带入

**实操技巧**：
- `/compact` 压缩当前会话上下文
- `/reset` 重置话题但保留长期记忆
- `memory-search` 精确查找记忆，而非暴力塞上下文
- 养成让 Agent 记忆重要信息的习惯（而非依赖长对话）

**模式价值**：与"有无论"相通——看"无"比看"有"更重要。减少冗余上下文，是让模型更聪明的关键。

## 核心模式四：错误自动修复 + 时间片管理（★★★★）

**来源**：知乎《OpenClaw智能Agent深度调教与2026全平台部署手册》

**核心观点**：长任务需要结构化保障，短周期输出是防止发散的核心手段。

### 错误处理链
```
遇到错误 → 判断能否自动修复 → 能修则立即执行 → 不能修则备选方案 → 无备选则上报选项
```
- 工具调用失败、文件读写异常、网络超时：无需用户指令，直接触发自动修复
- 修复失败后：简明上报错误类型、位置、建议方案

### 时间片规则
- 长任务自动切分为多个子任务，**单个子任务≤30秒**
- 每完成一个子任务输出进度提示，避免超时无响应
- 执行完成后汇总整体结果，保证流程可追溯

**模式价值**：这是"致人而不致于人"在执行层的体现——把失败路径和执行节奏提前结构化，不给Agent失控的空间。

## 核心模式五：权限最小化（安全原则）（★★★）

**来源**：Hostinger OpenClaw Best Practices

**核心观点**：权限按任务级别授予，不做默认全开。

**实操原则**：
- 新Skill先用 read-only 权限测试（不写入、不发请求）
- 不同任务分配不同权限：
  - 日志分析 → 只读特定目录
  - 报告生成 → 仅写一个输出目录
  - 数据摘要 → 禁止文件写入和外部请求
- 社区Skill在正式使用前审查代码，确认权限边界

**适用场景**：主要是VPS/生产环境。本地单用户场景优先级略低，但可在TOOLS.md中记录此原则。

## 反学习（不值得吸收的）

- ❌ 无脑安装大量 Skill（Skill 应设计成 SOP，不是能力清单）
- ❌ 用人格设定代替工作边界（少写"你是一个XX专家"，多写"什么情况下必须先问"）
- ❌ 多Agent分工（当前单用户场景，复杂度大于收益）
- ❌ Skill全权限上线（未测试就放开写入/网络权限）

### 新洞见一：记忆目录结构的具体设计（★★★★）

来源：《我的OpenClaw进阶配置：从"能用"到"好用"，5个模块让你的AI更懂你》

现有笔记只提了 MEMORY.md，未细化记忆子目录。这篇文章给出了可直接效仿的模板：

```
memory/
  MEMORY.md          # 核心索引：决策、偏好、持久事实
  projects.md        # 项目状态 + 任务追踪
  infra.md           # 基础设施配置速查
  lessons.md         # 问题记录 + 解决方案（错误日志）
  YYYY-MM-DD.md      # 每日对话日志
```

**何时写入哪类文件：**
| 内容类型 | 写入位置 |
|---|---|
| 决策 + 偏好 + 持久事实 | MEMORY.md |
| 项目进展 + 任务状态 | memory/projects.md |
| 踩过的坑 + 解决方案 | memory/lessons.md |
| 日常上下文 + 今日工作记录 | memory/YYYY-MM-DD.md |
| "记住这个"（临时/一次性） | memory/YYYY-MM-DD.md 或让Agent判断 |

**判断理由**：目录结构固化 = 减少"放哪"的认知负担，与"有无论"相通：结构清晰，模型就知道哪些该记、哪些不该记。目前我没有 memory/lessons.md，这条值得补。

### 16. memoryFlush：解决"AI失忆"的自动记忆维护机制

**来源**：[知乎 - OpenClaw进阶配置保姆级教程](https://zhuanlan.zhihu.com/p/2018266123171313527)

核心问题：会话变长时，Agent会"忘记"早期提到的关键信息，即使那些信息当时就在上下文里。

memoryFlush 是 OpenClaw 的自动记忆维护机制：
- 在长会话中自动将上下文里的关键信息固化到 memory 文件
- 让 Agent 在后续对话中能通过 memory-search 查询，而非依赖已丢失的早期上下文
- 减少因上下文窗口限制导致的重要信息丢失

**这与我已有的"固化习惯"互补**：我现在是手动告诉它"记住这个"；memoryFlush 是系统层面的自动机制，两者结合效果更好。

**模式迁移**：这是"把外部存储当大脑外延"的具象化——不依赖模型自身记忆力，而用文件系统和检索机制替代。

### 17. Skill链式工作流（Pipeline）：多Skill串联成端到端闭环

**来源**：[php中文网 - OpenClaw高级用法](https://m.php.cn/faq/2199953.html)

单一 Skill 做原子动作，真实复杂任务需要多步串联。Pipeline 允许将多个 Skill 按逻辑顺序自动编排：

**典型 Pipeline 示例**（每日简报）：
```
tavily-search（搜索热点）
→ summarizer（生成摘要）
→ obsidian/save（存入笔记库）
→ gog/send-email（推送邮件）
```

每一步的输出自动映射为下一步的输入（如 `results[0].content` → `text`），形成无人值守的端到端闭环。

**与多Agent的区别**：多Agent是"并联"，Pipeline是"串联"。Pipeline适合固定流程，Agent适合灵活决策。两者可叠加使用。

**以迂为直**：看似配置复杂，实际是把"每月花2小时做资料整理"变成完全自动化，省下的是长期时间。

---

### 18. 独立Session + 模型路由：彻底解决上下文污染

**来源**：[php中文网 - OpenClaw高级用法](https://m.php.cn/faq/2199953.html)

默认状态：所有任务共享同一上下文与模型实例 → 记忆污染 + 推理干扰。

解决方案（配置层）：
1. `session_mode: isolated` — 每个 Agent 有独立工作空间
2. 按任务类型指定专属模型：`search: gemini-3-pro`、`summarize: glm-5`、`coding: Opus`
3. 新 Agent 启动时加 `--session-id auto`

**结合多Agent独立MEMORY.md**：每个 Agent 既有独立上下文、又有专属模型路由，还有独立记忆文件，三层隔离彻底避免"上下文串台"。

**适合场景**：同时跑"育儿顾问"和"代码助手"两个 Agent，内容互不污染，各自专注。

### 新洞见：指令即工单——把聊天式指令翻译成结构化工单（★★★★★）

来源：人人都是产品经理《研究完OpenClaw源码后，我的10条使用原则》

**核心问题**：很多人习惯用"帮我看看这个东西""你研究一下"这类模糊聊天式指令，Agent容易过度解读，输出发散。

**正确做法**：把每次指令当成给Agent下一张工单，必须包含：
- **目标**（要解决什么问题）
- **输入材料**（只从哪些文件/来源读取）
- **执行步骤**（先做什么再做什么）
- **输出格式**（几段式、包含哪些字段）
- **停止条件**（不猜测、不泛泛总结）

**示例对比**：
- ❌ "帮我看懂这个项目源码"
- ✅ "只从README.md、ARCHITECTURE.md、src/auto-reply目录分析。先回答入口在哪，再回答消息链路。不猜测未读内容。输出5段式分析，含关键文件名和职责。"

**模式价值**：这与"致人而不致于人"最直接的结合点——指令模糊，则Agent掌控主动权；指令清晰，则人掌控主动权。

---

### 新洞见：检查点机制——高价值任务的半自动协作优于全自动（★★★★★）

来源：同上，原则八

**核心问题**：大语言模型存在幻觉，会虚晃一枪说"做完了"实际没做。

**操作方式**：
- 先让Agent列计划，不直接执行
- 中间结果先展示，确认后再继续
- 工具执行前先确认
- **Agent干完活后让它自己先检查一遍**（大幅减少幻觉率）

**适合场景**：写Skill、出客户方案、生成正式文档等高价值交付物。

**判断理由**：真正贴合"致人而不致于人"——全自动意味着放弃监督权，半自动才是掌握主动的正确姿势。目前luyu的任务（育儿推送等）都属于"定期稳定交付"类，这条直接适用。

---

### 新洞见：放大器思维——OpenClaw最适合做"研究放大"而非"终稿输出"（★★★★）

来源：同上，原则九

**三类真正高价值的用法**：
1. **产品研究**：文档扫描、模块梳理、关键文件抽取、问题定位
2. **多源杂乱内容预处理**：读会议纪要/访谈记录，先做结构化整理，再由人判断
3. **内容创作前的系统性调研**：拆结构、找证据、梳理观点，不直接写终稿

**不适合当**：搜索引擎替代、闲聊对象、一次性答案机器。

**模式价值**：这个认知框架让人重新校准对工具的期望值——不是AI替你做，而是AI帮你做好准备让你做得更好。与"以迂为直"相通。

### 新洞见：Git自动提交作为工作流规范（★★★★）

来源：知乎《OpenClaw智能Agent深度调教》、什么值得买《OpenClaw高级使用经验分享》

**核心问题**：当前 workspace 有大量 uncommitted 变更（learning/、memory/等），说明这个自动提交规则没有固化到日常操作流中。

**判断理由**：这解决了"AI重启后找不到之前做的工作"的问题。对于 luyu 的育儿推送等定时任务，如果某次执行更新了知识库或产生了新文件，自动 commit 确保下次执行时能读到历史状态。实操性强，直接可用。

### 新洞见：Session隔离的边界——workspace操作 ≠ ACP spawn隔离（★★★★）

来源：综合分析

**之前笔记写了"Session隔离"，但需要澄清边界**：

| 场景 | Session隔离状态 | 上下文共享 |
|------|--------------|-----------|
| `sessions_spawn` (ACP runtime) | ✅ 每次 spawn 独立 isolated session | 无共享 |
| Workspace 文件操作（read/write/exec） | ❌ 所有任务共享同一 Agent 实例 | 共享 |
| Cron isolated job | ✅ 独立 ephemeral session | 无共享 |
| 主会话（main session） | ✅ 独立上下文 | 无共享 |

**实操含义**：
- `sessions_spawn` 创建的子 Agent 是完全隔离的（独立 MEMORY.md、独立上下文）
- 但主会话中对 workspace 文件的修改，子 Agent 下次启动时也能读到（因为 workspace 目录是共享的）
- 这意味着：**workspace 目录是隐式共享上下文层**——文件层面的"记忆"跨 session 生效，但纯对话内容不跨 session

**应用**：如果某个 cron 任务需要"记住上次做到哪了"，写文件比依赖对话上下文更可靠（文件跨 session，对话不跨）。
